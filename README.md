# aspnet-ej1-demos — Legacy-to-Modern Migration Project

A real, hands-on Spec-Driven Development (SDD) migration of
[syncfusion/aspnet-ej1-demos](https://github.com/syncfusion/aspnet-ej1-demos) (1147-page
ASP.NET Web Forms app) using [OpenSpec](https://github.com/Fission-AI/OpenSpec) and a
custom Claude Code skill (`aspx-analyzer`). Two capabilities are built and genuinely
browser-verified so far; see `example-walkthrough/` for the full real transcript.

**This repo ships only what we authored** — the legacy vendor source is not committed
here (it's a public clone, re-fetchable anytime; see below). What's in git: the
OpenSpec workspace, the modern stack, the skill, and the full documented process.

**New to OpenSpec?** [OPENSPEC_GUIDE.md](OPENSPEC_GUIDE.md) is a standalone, generic
guide — install, every CLI command in one line with when to use it, the day-to-day
SDD loop, and a legacy-migration workflow, all distilled from what actually happened in
this project.

**Need the precise technical reference?** [OPENSPEC_HANDBOOK.md](OPENSPEC_HANDBOOK.md) —
execution model (what runs when: OpenSpec CLI vs. the skill vs. Claude Code), Claude Code
Agent Skill integration, and the complete command surface for both OpenSpec and
`aspx-analyzer`.

## Getting the Legacy Source

Not tracked in this repo (see `.gitignore` and "Why legacy/ isn't in git" below). Clone
it into `legacy/` before running anything that reads from it:

```bash
git clone --depth=1 https://github.com/syncfusion/aspnet-ej1-demos.git legacy
rm -rf legacy/.git   # optional — keeps legacy/ a plain vendored folder, not a nested repo
```

On Windows, if the clone fails with a "Filename too long" error partway through (a known
issue with this specific repo's deeply-nested `ReferenceAssemblies/` folder), use:

```bash
git -c core.longpaths=true clone --depth=1 https://github.com/syncfusion/aspnet-ej1-demos.git legacy
```

and clone into a short path if it still fails (see `example-walkthrough/EXAMPLE_WALKTHROUGH.md`
Part 1 for the exact failure this project hit and how it was resolved).

## How OpenSpec + the aspx-analyzer Skill Integrate Here

This project is the skill and the CLI working as one pipeline, not two separate tools
glued together after the fact. Concretely, in this repo:

```
1. skill/scripts/aspx_analysis_skill.py       legacy/ (Web Forms source)
        │                                            │
        ▼                                            ▼
   parses every .aspx/.ascx/.master/web.config, builds ONE JSON index
        │
        ▼
   example-walkthrough/analysis-output/aspnet-ej1-demos_aspx_index.json
        │
        ├──────────────────────────────┐
        ▼                              ▼
2. aspx_roadmap_emitter.py       aspx_openspec_emitter.py
   → MODERNIZATION_ROADMAP.md    → modernization/openspec/config.yaml
     (tech-stack setup +           (tech-stack context + modernization rules,
      capability build order,      auto-injected into every future
      simplest/lowest-risk          `openspec instructions` call — verified
      first)                       in EXAMPLE_WALKTHROUGH.md Part 12.3)
                                  → modernization/openspec/changes/<area>/
                                     proposal.md (one stub per functional
                                     area/capability — 11 exist, unused so far)
        │
        ▼
3. openspec new change modernize-<capability>       (the OpenSpec CLI itself, from here on)
   openspec instructions proposal|design|specs|tasks --change <name>
        │
        ▼
   author each artifact (proposal.md → design.md → specs/*/spec.md → tasks.md)
        │
        ▼
4. Implement strictly per tasks.md — this project's two built capabilities
   (modernize-grid-checkbox-selection, modernize-grid-basic-selection) both
   landed in modernization/OrdersApi/ + modernization/orders-grid-web/ this way,
   never by freehand coding outside what a task line specified.
        │
        ▼
5. openspec validate <name> --strict   → openspec archive <name>
   (step 5 not yet reached for either capability here — see Status table)
```

**Where each piece lives right now:**

| Artifact | Path |
|---|---|
| Legacy analysis index (frozen snapshot) | `example-walkthrough/analysis-output/aspnet-ej1-demos_aspx_index.json` |
| Modernization roadmap (frozen snapshot) | `example-walkthrough/analysis-output/MODERNIZATION_ROADMAP.md` |
| OpenSpec config (tech-stack context + rules) | `modernization/openspec/config.yaml` |
| In-flight + stub changes | `modernization/openspec/changes/*/` |
| Durable specs (fills in on first archive) | `modernization/openspec/specs/` (empty so far) |
| The skill itself | `skill/` |

**Why this split matters in practice**: `openspec/config.yaml`'s auto-injected context
means every future `openspec instructions proposal --change <new-capability>` call
already knows this is ASP.NET Web Forms migrating to ASP.NET Core + React, already knows
the direct-SQL page count, already knows to require legacy page references in every
proposal — without re-typing any of that. That's the entire point of running the skill's
emitter before starting OpenSpec changes, verified working end-to-end in
`example-walkthrough/EXAMPLE_WALKTHROUGH.md` Part 12.3.

## Using This Project

### Continue the OpenSpec workflow

```bash
cd modernization
openspec list                                    # see all in-flight changes
openspec status --change modernize-grid-basic-selection
openspec validate modernize-grid-basic-selection --strict
```

Full command reference and day-to-day loop: `skill/SKILL.md`, and the parent
`aspx-analysis-skill` tool repo's `openspec_setup.md` (a separate repo — general
OpenSpec+skill reference, not specific to this project).

### Run the frontend

```bash
cd modernization/orders-grid-web
npm install
npx json-server mock/db.json --port 3001   # local mock API (no .NET SDK needed)
npm run dev                                 # then open http://localhost:5173
```

### Build the backend (needs a real .NET 8 SDK)

```bash
cd modernization/OrdersApi
dotnet build
dotnet run
```

### Re-analyze the legacy repo

Requires `legacy/` cloned locally first (see above — it's gitignored). Writes to
`example-walkthrough/analysis-output/` explicitly so the output doesn't land inside the
gitignored `legacy/` folder by default:

```bash
python skill/scripts/aspx_analysis_skill.py legacy --view project --workers 8 \
    --save-report --output example-walkthrough/analysis-output
python skill/scripts/aspx_roadmap_emitter.py \
    example-walkthrough/analysis-output/legacy_aspx_index.json \
    --output example-walkthrough/analysis-output
```

### Use `skill/` as a live Claude Code skill elsewhere

`skill/` is a plain copy of the Agent Skill — Claude Code only auto-discovers skills
under `.claude/skills/<name>/`. To activate it in another repo (or this one):

```bash
mkdir -p .claude/skills
cp -r skill .claude/skills/aspx-analyzer
```

## Why `legacy/` Isn't in Git

It's a public, re-clonable vendor repo we don't modify — committing it here would mean
carrying ~1.4 GB of someone else's source (plus their NuGet `packages/` restore and
`ReferenceAssemblies/`) for zero benefit, since anyone can `git clone` the original in
seconds. This repo tracks only what was actually authored: the OpenSpec workspace, the
modern stack, the skill, and the documented process. See `.gitignore`.
